## Assumptions
- You have an Android test device (or emulator) connected via USB with USB debugging enabled.
- The app is already installed on the device.
- I’ll use a placeholder `com.example.testapp` for the package name (replace with your actual package).
- You own the app and are authorized to analyze it.

## Commands / Code

```bash
# 1. List all installed packages to confirm your app’s package name
adb shell pm list packages | grep <keyword>

# 2. Get the APK path(s) for the app
adb shell pm path com.example.testapp

# 3. Pull the APK to your local machine
adb pull /data/app/com.example.testapp-*/base.apk testapp.apk

# 4. Check the APK’s hash and file type (optional)
sha256sum testapp.apk
file testapp.apk

# 5. Decompile with apktool (resources and smali)
apktool d testapp.apk -o testapp_apktool

# 6. Convert to Java with jadx (easier to read)
jadx -d testapp_jadx testapp.apk

# 7. (Optional) Use jadx-gui for interactive browsing
jadx-gui testapp.apk
```

---

## What each command does

| Command | Purpose |
|---------|---------|
| `adb shell pm list packages` | Lists all packages; pipe to `grep` to find your app. |
| `adb shell pm path <package>` | Returns the full path of the APK file on device. |
| `adb pull ...` | Copies the APK from device to your computer. |
| `sha256sum` / `file` | Verify integrity and confirm it’s a valid APK archive. |
| `apktool d` | Decompiles the APK into Smali code, resources, and AndroidManifest.xml. Useful for checking manifest permissions, resource strings, and hardcoded URLs. |
| `jadx -d` | Decompiles the APK directly to readable Java source code. Best for finding hardcoded endpoints and insecure storage logic. |
| `jadx-gui` | Opens a graphical interface to browse decompiled sources, search strings, and navigate easily. |

## Next safe steps (analysis guidance)

1. **Look for hardcoded endpoints**  
   In the decompiled Java (jadx output), search for:  
   - `http://` or `https://` → check for hardcoded API URLs  
   - IP addresses (e.g., `10.0.2.2`, `192.168.`)  
   - Strings like `api_key`, `secret`, `token`, `password`  
   Use `grep -r "https\?://" testapp_jadx/` or jadx-gui’s text search.

2. **Inspect insecure storage**  
   - **SharedPreferences**: Search for `getSharedPreferences`, `commit()`, `apply()`. Check if sensitive data is stored without encryption.  
   - **SQLite databases**: Search for `openOrCreateDatabase`, `SQLiteOpenHelper`. See if the database file is world-readable (check `MODE_WORLD_READABLE` in the manifest or code).  
   - **File I/O**: Look for `FileOutputStream`, `Context.getFilesDir()`, `getExternalFilesDir()` – especially for storing tokens or user data in plaintext.  
   - **Logging**: Search for `Log.d`, `Log.i`, `Log.w` – insecure logs can leak credentials or tokens.

3. **Check AndroidManifest.xml** (from apktool output)  
   - `android:allowBackup="true"` → backups can expose data.  
   - `android:debuggable="true"` → debug builds are more vulnerable.  
   - Permissions: look for `INTERNET`, `READ_EXTERNAL_STORAGE` – are they necessary?

4. **Use static analysis tools** (optional)  
   - `qark` or `MobSF` can automate many checks, but manual review is recommended for learning.

5. **Never** attempt to run dynamic analysis (e.g., bypass SSL pinning, hooking) unless you are in a controlled lab environment and you have explicit permission.
